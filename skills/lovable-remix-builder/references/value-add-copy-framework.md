# Value Add Copy Framework

Use this when generating homepage or landing-page copy for any app, not just the current niche.

## Core Rule

Do not write copy that only names features.

Write copy that makes the user understand:
1. what gets better
2. why it gets better
3. what the app is doing differently
4. why they should trust it enough to act now

## The Copy Spine

### 1. Promise
Write the clearest valuable outcome first.

Prefer:
- who it is for
- what improves
- what the app helps them avoid

Good pattern:
`[Audience] gets [better outcome] because the app does [specific mechanism].`

### 2. Mechanism
Explain what the app actually changes.

Users should understand:
- what inputs the app uses
- what work it removes
- what signal or leverage it adds

If the product uses AI, filters, automation, ranking, memory, or workflow logic, make that concrete.

### 3. Payoff
Translate the mechanism into practical value:
- less wasted time
- fewer mistakes
- faster insight
- stronger output
- calmer workflow
- higher quality decisions

### 4. Proof Or Plausibility
Even without hard metrics, show why the claim is believable:
- a specific workflow
- before/after contrast
- a compatibility or recommendation example
- a short “why this works” panel

### 5. Objection Handling
Answer the quiet questions:
- Is this hard to start?
- Will this work in my city, team, or workflow?
- Is this private, secure, or trustworthy?
- Do I need to do a lot of setup?

### 6. CTA Ladder
Not every visitor is ready for the same step.

Prefer:
- one primary CTA
- one lower-friction supporting CTA

Examples:
- `Start free` + `See demo`
- `Join waitlist` + `Check your region`
- `Book a call` + `View sample output`

## Section-Level Best Practices

### Hero
- one strong promise
- one supporting sentence explaining the mechanism
- one primary CTA
- one supporting CTA
- 3 short proof bullets at most

### Problem / Why Switch
- contrast the old way with the improved way
- explain the cost of not changing
- avoid melodrama; stay concrete

### How It Works
- describe the flow in 3 steps or beats
- tie each step to user value, not internal implementation only

### Feature Grid
- each card should answer `why do I care?`
- rename vague features into user outcomes when possible

### Proof / Comparison
- use side-by-side contrast, result snapshots, metrics, or testimonials
- make the improvement feel visible

### Final CTA
- restate the promise in simpler language
- reduce friction
- give one clear next step

## Anti-Patterns

Avoid:
- “all-in-one platform” with no specifics
- “streamline your workflow” without saying what changes
- feature dumping
- empty adjectives like “seamless,” “revolutionary,” or “powerful” without proof
- three equal CTAs competing for attention

## Quick Test

Before shipping copy, ask:
- Can the visitor say what the app helps them do?
- Can they explain why this app is better than a generic alternative?
- Does each section add new information instead of repeating the headline?
- Is the CTA ladder clear?
